## Discoveries
- **Rendu de contenu LLM = webview PRINCIPALE, pas l'iframe sandboxée** : la CSP `main` n'a pas de `default-src` → toute directive absente est non restreinte. Le « 0 réseau » d'une réponse LLM repose donc sur **DOMPurify seul** → **allowlist stricte** obligatoire (pas un denylist : `<video>/<audio>/<source>/<svg><use>` sont des beacons qu'un `FORBID img` laisse passer). Le tag `<style>` interdit ne retire pas l'attribut `style="background:url()"` → hors `ALLOWED_ATTR`.
- **Ollama /api/chat vs /api/generate** : pour le dialogue multi-tours, `/api/chat` (rôles system/user/assistant) évite la dérive de complétion d'un prompt concaténé single-turn (le modèle « complète » de faux tours).
- **Rendu Markdown par token = O(n²) + flicker** (table à moitié, ``` non fermé) → afficher la bulle en **texte brut pendant le stream**, parser+sanitize **une seule fois à la fin**.
- **Svelte 5 `$state` profond** : muter la **ref locale** poussée dans un `$state[]` ne re-render pas ; muter via l'**index** (`arr[idx].content += t`) passe par le proxy → réactif.
- **Anti-TOCTOU d'un envoi** : poser `generating=true` + le contrôleur d'abort **avant tout `await`**, sinon deux envois rapprochés s'entrelacent (bulle partagée, contrôleur écrasé).
- **`$effect` d'ouverture de vue** : la vue chat ne doit JAMAIS lire `copilot.models` au montage (sinon spawn Ollama au boot si le panneau est persisté ouvert) — décision onboarding/empty gatée sur l'intention explicite (entrée vue Modèles / envoi).
- **Troncature silencieuse à DEUX couches** (14.2/14.3) : segmenter ne suffit pas — un prompt plus long que `num_ctx` est tronqué **à gauche par Ollama** (défaut 2048/4096). Il faut **fixer `num_ctx`** ET dimensionner les segments dessous. Piège : segmenter en **caractères** alors que le budget est en **tokens** (CJK ≈ 1 tok/car vs latin ≈ 0,25) → garder `SEGMENT_CHARS ≤ num_ctx − marge`.
- **Résumé lent = map-reduce déclenché trop tôt + petit modèle bavard** : seuil trop bas → N passes séquentielles sur CPU. Fix : fenêtre généreuse (une note tient en 1 passe) + `num_predict` borné sur la phase map. Le plancher reste le modèle (0.5b, CPU pur, NPU non exploité).
- **Packaging release Ollama = résolution EXE-RELATIVE** : `ollama.exe` charge `lib/ollama/` (llama-server + DLLs) **relativement à son propre dossier**, pas via `OLLAMA_LIBRARY_PATH` (no-op sur 0.32). `bundle.resources` place `lib/ollama/` frère de l'exe sous `resource_dir()` → OK. Prouvé sans installer : générer via le `ollama.exe` de `target/release`.
- **Ancrage d'un petit modèle** : il dérive vers ses connaissances internes malgré la consigne système. Leviers efficaces : **température basse** (~0,2) + **rappel d'ancrage collé à la question** (proximité de génération = plus de poids). Atténue sans éliminer → vrai fix = meilleur modèle.
- **Q4_0 = LA quantification sur ARM** : llama.cpp repacke les poids **Q4_0** au chargement vers des kernels KleidiAI/i8mm (PR #9921 : **×2,3 décodage, ×2,9 prefill**, confirmé sur Snapdragon X). Piège : **les tags Ollama par défaut sont en Q4_K_M** (non repacké) — vérifié au digest : `qwen2.5:3b` = `:3b-instruct-q4_K_M`. Toujours des tags **`-q4_0` explicites** sur cette machine.
- **Les params « E » de Gemma (E2B/E4B) n'économisent que la VRAM GPU** : sur CPU, TOUS les poids se chargent en RAM → E4B ≈ 6,9B en calcul (10,7 t/s, prefill 45 s mesurés), E2B = 4,5B/4,3 Go. Un **vrai 3-4B dense est plus léger ET plus rapide** que « E2B ».
- **Falaise des ~3B pour la reformulation** : sous 3B, l'écho du prompt (recopier les consignes) est structurel (MT-Bench-French : 2,34-4,37 sous 3B vs 5,86-6,08 à 3B ; arXiv 2510.13935). Aucune ingénierie de prompt ne fiabilise un 0.5-1.5B pour cette tâche.
- **Thinking Ollama** : ON par défaut quand le modèle le supporte (Qwen3, SmolLM3) ; `think:false` sur /api/chat//api/generate le coupe, MAIS **l'envoyer à un modèle non-thinking = erreur API** (ne pas hardcoder) ; aucun PARAMETER Modelfile. Qwen2.5/Gemma/Llama 3.2 = non-thinking par nature.
- **llama3.2:3b faible en français malgré MT-Bench correct** : IFEval FR = 34,46 (fond du leaderboard fr-gouv) — un score EN ne prédit pas l'obéissance de format en FR.
- **dspark (DeepSeek) = drafter de décodage spéculatif, pas un modèle** : `dspark_qwen3_4b_block7` = 5 couches lisant les états cachés d'un Qwen3-4B cible (block7 = blocs de 7 tokens) ; gain GPU (SGLang/MLX), **rien sur CPU/Ollama**.
- **NPU Hexagon : prêt côté machine, bloqué côté Foundry Local** : pilote 30.0.220.3000 conforme, mais le provider QNN ne s'enregistre pas (bugs ouverts #393/#259, pas de commande d'install manuelle). Et en **décodage le NPU perd contre le CPU Q4_0** (19,5 vs 36,3 t/s sur Phi-4-mini) — son vrai gain = prefill (~18×). Nexa/OmniNeural exclus : activation en ligne = casse le 0-réseau.
- **« RAM libre » Windows est trompeuse** : les ~2 Go « libres » excluaient le cache fichier ; 16 Go total → un 3-4B Q4 tient large, l'OS libère le cache à la demande.
- **`ConvertTo-Json` PowerShell détruit le ledger** : il réécrit TOUT le fichier (61 → 404 lignes + échappements `>`). Mutation du ledger = **édition chirurgicale de la seule ligne** (Edit), puis validation `ConvertFrom-Json`.
- **Barre de pull Ollama multi-couches** : chaque blob a son propre `total` → un % par couche boucle 0→100 ; agréger par `digest` (Σ completed / Σ total).

## Decisions
- **Copilote = panneau `aside` DROIT** (« Doku-San »), pas la vue sidebar gauche initialement supposée ; la gestion des modèles (13.4) y est relocalisée. Maquette Claude Design suivie au pixel (décision A).
- **« Coquille visuelle maintenant, câblage plus tard »** (décision B) : doc courant seul en v2.0 ; puces `+ Contexte` multi-docs = coquille désactivée → Epic 15 (RAG).
- **Découpage 14.0 / 14.1** : coquille+chrome+relocalisation (14.0) séparés du chat (14.1) pour ne pas laisser 14.1 gonfler en silence (leçon rétro S11).
- **13.5 créée** hors décomposition pour solder au prochain `tauri build` les 2 dettes reportées (packaging `lib/ollama` release + association `.pdf`).
- **Vélocité recalibrée** : sprints d'1 semaine / 6-8 stories (S11 = 4×M en 1 jour vs 2 semaines planifiées).
- **Débit `t/s` (carte modèle actif)** non affiché en 14.0 : refus d'un chiffre inventé (mesuré plus tard) ; seule stat réelle = disque (+ params dérivés du nom si parseables).
- **Lecture complète d'un gros doc unique → RAG (Epic 15)**, pas « agrandir la fenêtre » (repli qui plafonne au max du modèle et ralentit chaque question) : décision utilisateur ; en v2.0 on lit jusqu'au seuil et on **signale honnêtement** (badge). Mode « document courant » à prévoir en 15.3.
- **13.5 = dette de PREUVE, 0 code** : config packaging déjà correcte ; validé par build + génération réelle depuis `target/release` (chemin `resource_dir()` exercé) + app installée.
- **Upgrade modèle** : cible privilégiée **Gemma 3n E2B (voire E4B si pas trop lent)** — pensé edge/NPU, cohérent Snapdragon X. Reporté ; banc d'essai = le cas Keycloak→Zitadel (dérive d'ancrage). *(Tranché le soir même, voir ci-dessous.)*
- **Modèle copilote FINAL (nuit)** : philosophie « **gadget discret** » actée par l'utilisateur — le copilote ne doit jamais occuper la machine. Défaut = **`qwen2.5:1.5b-instruct-q4_0`** (935 Mo, non-thinking, Q4_0 repacké ARM) ; **`qwen2.5:3b-instruct-q4_0`** (1,8 Go) installé pour la qualité en 1 clic. Verdict issu d'une deep-research (104 agents, 21 claims confirmées 3-0). Rejetés en test réel : E4B (lent), E2B (lourd), Qwen3/SmolLM3 (thinking), llama3.2 (IFEval FR 34,46), gemma3:4b (pas de Q4_0 dans le budget).
- **Epic 17 « Backend NPU » créé et PARQUÉ** : Foundry Local (ONNX/QNN) = le seul chemin NPU compatible 0-réseau, mais provider QNN bloqué par bug amont → d'abord épuiser le levier CPU+Q4_0 ; re-tester à une future version (guetter microsoft/foundry-local#393). OmniNeural/Nexa exclus définitivement (anglais + activation en ligne).
- **16.1 validée natif** → ledger **56/57**. Reste 16.2 (dernier stretch).
